Introduction

The MFC Doc/View base purports to separate the details of programming the data in the app from the details of programming the man/machine output to the various devices (e.g. display and printers) in the app. It does this by sending calls in a specific order to the app's View class.

Every app has very specific needs but they may differ among the applications one writes. I finally realized that I was reinventing the wheel everytime I started a new application. So I created MakeApp, an application which will create a "starter" application. When the app is first created it is ready to run with sample commands and code of various kinds already implemented. A "#define" controls some of the examples and other examples are just there to use as needed.

Unlike the old c apps that start with main, MFC Doc/View apps start with a subclass of CWinAppEx. Eventually during the initiation of the app, two additional objects are created, a doc object and a view object. They work together to all an app to function.

It turned out that all apps need some things done in each MFC app opbject. So I wrote CApp, CDoc and CScrView. So the three project objects are subclasses of CApp, CDoc and CScrView. CApp provides an identity of the App to Windows, and pointers the doc, view and mainframe. CDoc provides features for reading and writing to files including implementing a replacement for serialize. CScrView provides a framework for managing the output to a display and printer. CScrView is a subclass of CScrollView so CScrView also helps manage the window scroll bar.

Output to the window starts with an invalidate call to windows. An invalidate may occur naturally, for example when the size of the window changes, or explicitly when the app executes an invalidate call. Then an OnPrepareDC call is executed. It contains pointers to a device context and a print info objects. After initialing things and it returns then OnDraw is called. OnDraw is responsible for "drawing" on the window.
